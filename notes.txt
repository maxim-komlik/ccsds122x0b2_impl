Finally got it!!!!
Well, there is a gaggle that contains of 16 blocks. Each block has a set of words computed. Starting 
length of all words is well-known (It is defined in table). At every stage a block might have some words 
of length 2, 3, or 4, but throughout the process of decoding all particular lengths for every word are 
known. Another block in the same gaggle may have the same words of different lengths. But anyway, there
are only 3 word length options available. The gaggle parameters define one coding option for every 
word length for all words of that length for all blocks within the gaggle, that is, supposing that b1 has
transH word 4-bit len and types[C] word 3-bit length, and b2 has transH word 3-bit len and types[C] word 
4-bit length, the same coding option for transH and types[C] 4-bit words must be used, as well as the same
coding option for types[C] and transH 3-bit words must be used. So 4 bits for coding option specification 
will be used totally, assuming there is no words of len 2 in the gaggle.
Each word that is computed over a block is 4-bit lenght at most (and probably is 4-bit at the start). It 
seems that grandchild generation was further partitioned into subgroups to match this pattern for them as 
well. So in fact the sequence of words coded is known, but 3-bit words are coded in place with 3-bit 
entropy code as needed, it may be followed by 4-bit word that is coded in-place with 4-bit entropy encoding
and so on. The words of different length are interleaved.

 BitDepthAC_Block

	0 if positive (nonnegative)

if x is of type 0 or 1:

	1 if negative

Steps to prepare for bitplane encoding:
1) pack bitmap contents into blocks
2) build segments
3) compute BitDepthDc and BitDepthAc
4) compute quantized DC coeffs for each segment
5) compute quantized DC diffs
6) map quantized DC diffs
* write to blocks non-quantized part of DC

Segment Pre-coder receives packed bitmap buffers computed by ForwardWaveletTransformer; the lines in these 
buffers are alligned, and necessary alignment offsets are allocated at both beginning and ending of the 
line (among not used part of alignment unit at the end of the data). Length of the image line and segment 
size are indpendent values, therefore there may be data of any number of segment in a line of DC coefficients 
(keepeing in mind a limitation that minimal size of the segment is set to 16 blocks). Therefore several 
scenarios are possible when analyzing mutual layout of the bitmap row and set of DC coeffeicients for 
the same segment:
! - segment boundary
| - row boundary
_ - data values
. - auxilary memory space (e.g. for alignment)
1) segment is alligned properly, the whole segment resides in the row:
...|!____________!______________|........|_____  //
...|_______!______________!_____|....
           ^ alligned memory address

2) segment is not alligned properly, the whole segment resides in the row:
...|_____!_____________!________|....
         ^ not alligned memory address

3) segment is alligned properly, segment resides in several rows:
...|____________________!_______|........|__________!____  //
                        ^ alligned memory address
...|____________________!_______|........|___//___|........|____!____  //
                        ^ alligned memory address. Segment resides in 2 or more rows
						
3) segment not alligned properly, segment resides in several rows:
...|____________________!_______|........|__________!____  //
                        ^ not alligned memory address
...|____________________!_______|........|___//___|........|____!____  //
                        ^ not alligned memory address. Segment resides in 2 or more rows

Requirements:
	To compute bit depth:		not necessarily alligned, auxilary data is zeroed (including beginning and 
ending of the first and the last allign units), not necessarily continuous. (Here is an option to overwrite 
the data of the next or the previous segments temporarily and revert the changes after operation)
	To compute DC diff values:	
We need segment DC data to be alligned, continuous and consequtive to be processed properly with 
vectorization applied.
We may pack blocks first and then free unnecessary bitmap buffers (all except DC); then process DC 
coefficients. This way we can compute BitDepthAc first over segment block by block first, and then use 
the result value to process DC coefficients, possibly getting rid of unnecessary (already processed) data.



Transfomer class has extension buffers for every (x3) level vertical pass. The buffers contain full-width
input data rows for different levels, the buffer varies in length according to level input row len. Totally 
6 buffers needed, each buffer is half-len of input data row. Buffers are overwriten on the fly according to 
current transform frame position.

General call for image tranform performs image boundary extension. Subsequent transform level calls need to 
extend boundaries on demand near the source image boundaries, however, extension is performed on partially 
transformed data, therefore it cannot be performed before level-1 call.
Transform level calls are not aware of current frame position, so special signaling scheme should be 
introduced to leverage on-demand extension.
Can be implemented as 2 boolean flags: for begining and for ending.

Vertical pass calls are aware of current frame position.
Vertical pass procedure always extends high boundary (begining). Low boundary (ending) extension is required 
once the bottom of the image is reached.
Can be implemented as a boolean flag for image bottom boundary.
